# Звіт: Лабораторна Робота №5
**Оркестрація ML-пайплайнів: Від CI/CD до Continuous Training**

---

## 1. Мета роботи

1. Опанувати принципи Workflow Orchestration за допомогою **Apache Airflow**.
2. Навчитися створювати оптимізовані Docker-контейнери для ML-задач (multi-stage builds).
3. Реалізувати автоматизований ML-пайплайн (DAG), що охоплює підготовку даних, тренування та реєстрацію моделі.
4. Впровадити CI для перевірки цілісності DAG та успішності збірки Docker-образу.

---

## 2. Реалізація

### 2.1 Контейнеризація (`Dockerfile`)
Реалізовано **multi-stage build** для зменшення розміру фінального образу:
*   **Stage 1 (builder)**: Встановлення `build-essential` та всіх Python-залежностей.
*   **Stage 2 (final)**: Використання `python:3.11-slim`. Копіювання лише необхідних бібліотек та коду проекту. Це забезпечує легкість образу та швидке розгортання.

### 2.2 Оркестрація з Apache Airflow (`docker-compose.yaml`)
Налаштовано локальне середовище Airflow, яке:
*   Використовує `LocalExecutor` та PostgreSQL як базу метаданих.
*   Монтує папку з проектом у контейнер (`/opt/airflow/ml_project`), що дозволяє Airflow керувати DVC та запускати скрипти тренування.

### 2.3 Розробка пайплайну (`dags/ml_pipeline.py`)
Створено DAG `ml_training_pipeline`, який складається з таких етапів:
1.  **wait_for_data**: PythonSensor, що очікує на появу вхідних даних.
2.  **data_preparation**: Запуск `dvc repro prepare` для підготовки ознак.
3.  **model_training**: Тренування моделі за допомогою `src/train.py`.
4.  **evaluate_and_branch**: BranchPythonOperator, що перевіряє `metrics.json`. Якщо F1-score вище 0.3, пайплайн переходить до реєстрації.
5.  **register_model**: Реєстрація моделі в MLflow Model Registry.
6.  **stop_pipeline**: Завершення роботи, якщо якість моделі недостатня.

### 2.4 Автоматизація CI (`.github/workflows/main.yaml`)
Додано новий workflow для Lab 5, який перевіряє:
1.  **DAG Integrity**: Тест `tests/test_dag.py` перевіряє, чи завантажується DAG без помилок імпорту та чи правильно визначені ID.
2.  **Docker Build**: Перевірка, що `Dockerfile` збирається без помилок.

## 4. Висновки

1. **Docker Multi-stage** дозволяє відокремити процес побудови залежностей від середовища виконання, що є критичним для ML-задач з важкими бібліотеками.
2. **Airflow** забезпечує прозорість та надійність виконання складних пайплайнів, дозволяючи легко реалізувати логіку розгалуження (branching) на основі метрик.
3. **CI для DAG** є важливим етапом, що запобігає "падінню" оркестратора через помилки в коді самого пайплайна ще до його розгортання.
